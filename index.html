NIELSONS RECOMMENDED HEURISTICS

Visibiltiy Of System Status Match Between System & Real World User Control & Freedom Consistancy & Standards Error Prevention Recognition Rather Than Recall Flexibility & Efficiency Of Use Aesthetic & Minimalist Design Recognise, Diagnose, & Recover from Errors Help & Documentaion

VISIBILITY OF SYSTEM STATUS

Nielson's first heuristic : the system should always keep users informed about its current state. When a user is aware of system states, he can actively change them, hence imparting him with control.

To keep the user up to date on its status, a system should also be able to provide immediate feedback to a user's actions. When I press a button, I expect the button to tell me that it has been pressed. Deprived of responsiveness, a user will end up feeling helpless and frustrated.

MATCH BETWEEN SYSTEM & REAL WORLD

Nielson's second heuristic : the system should talk to the user in layman's terms, not in technical ones. Complicated language repels, understandable language attracts. If you have no choice but to use technical acronyms, explain them - don't make users leave the page to Google.

National Aeronautics Space Administration
Self Contained Underwater Breathing Aparatus
Automated Teller Machine
Vital Technical Marketing

Leverage a user's familiarity with real world objects and models. If there is an online representation of a physical object, I will know immediately how to use it. A foreign, digital view, however, may elicit confustion. Smoothen the user's transition from off to online.

USER CONTROL & FREEDOM

Nielson's third heuristic : hand over complete control to the user. If the user wants to leave the website, let him. If the user wants to terminate an important process, let him. The exit sign should be clearly visible, like it is in physical buildings.

CONSISTENCY & STANDARDS

Nielson's fourth heuristic : do not make users remember different things for different situations. If the exit button is red on one page, it should be red on all the others. Stick to one of the below, not all.

ERROR PREVENTION

Nielson's fifth heuristic : do not blame or criticize users when they make a mistake. Instead, prevent errors from occurring in the first place. Include constraints so that users do not have much room to err; offer suggestions whenever an action is required of the user; provide smart defaults in appropriate instances; and be flexible when receiving user input. An ommision of a space or not in their phone number shouldn't cause an error window to pop up.

If an error does occur, however, guide the user in clear steps. Never flood them with technical jargon or offer useless advice.

RECOGNITION RATHER THAN RECALL

Nielson's sixth heuristic : do not drown the user's brain. Keep all pertinent information visible to the user and ensure that they see instructions, not recall them.

FLEXIBILITY & EFFICIENCY OF USE

Nielson's seventh heuristic : tailor your system for both novice and expert users. Experts should be able to quickly jump around using keyboard shortcuts, while novices can always opt to take the slower, more visual path.

AESTHETIC & MINIMALIST DESIGN

Nielson's eighth heuristic : we have short memory spans. Every piece of instruction, every line of text, should be short and concise. More words will dilute the core information.

HELP USERS RECOGNISE, DIAGNOSE, & RECOVER FROM ERRORS

Error messages should be expressed in plain language (no codes), precisely indicate the problem, and constructively suggest a solution.

HELP & DOCUMENTATION

Even though it is better if the system can be used without documentation, it may be necessary to provide help and documentation. Any such information should be easy to search, focused on the user's task, list concrete steps to be carried out, and not be too large.
